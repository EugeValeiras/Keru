# ADR-001 -- Adopt Dapr for inter-service invocation and pub/sub

| | |
|---|---|
| **Id** | ADR-001 |
| **Status** | Accepted |
| **Date** | 2026-05-26 |
| **Deciders** | Architecture Team |
| **Architecture Release** | arch 1.0.0 (Current) |

> Worked example exercising `../../../../contracts/adr-contract.md`. A binding ADR:
> it promotes a runtime choice (normally implementation territory) into an
> architecture constraint per Architecture Supremacy.

## Context

The closed architecture forbids synchronous Manager-to-Manager calls (R-03), yet
Managers must communicate: `ChargeSessionManager` must hand a completed session
to `BillingManager`, and `OverstayManager` must reach `ALPRManager`. The
decoupling residue from S-03 (auth/billing independence, C-03) only pays off if
the cross-Manager link is asynchronous and additive -- it is what let S-15 (AFIR)
be absorbed ten years later without rewriting `BillingManager` or
`ChargeSessionManager`.

## Decision

Use **Dapr** building blocks for inter-service communication: pub/sub for
cross-Manager events, service invocation, and the state building block where a
Manager needs durable workflow state.

## Binding scope

| Field | Value |
|---|---|
| **Binding** | Yes |
| **Claims** | Inter-service (Manager-to-Manager) invocation + pub/sub runtime + state building block. |
| **Constrains** | All Managers (`ChargeSessionManager`, `ALPRManager`, `BillingManager`, `OverstayManager`, `LegalCaseManager`). |
| **Conformance check** | The plan's `Manager -> Manager` paths go through Dapr pub/sub; no synchronous Manager call appears. (Constitution Check question 6.) |

## Consequences

- Cross-Manager communication is async, additive, and observable -- new
  consumers (e.g. AFIR's payment flow) subscribe without touching producers.
- Dapr's actor model aligns with the actor-style grouping (R-25: one Manager =
  one service), and its state building block keeps Engines stateless.
- The implementation half may not substitute a synchronous RPC mesh or a direct
  message library for the claimed concerns -- that would violate Supremacy.

## Alternatives considered

- **Synchronous service mesh (gRPC) for Manager-to-Manager** -- rejected: it
  re-introduces synchronous Manager coupling (R-03 violation) and would have made
  AFIR invasive.
- **A bespoke in-process event bus** -- rejected: does not survive the per-country
  topological split (S-13 / C-04) where Managers may run in different units.
