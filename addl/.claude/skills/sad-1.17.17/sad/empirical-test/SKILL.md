# empirical-test (S6)

Produces the Empirical Test fragment of a SAD: the Residual Index `Ri = (Y - X) / S` test, where X is the count of test stressors survived by the naïve architecture, Y is the count survived by the residual architecture, and S is the test set size. The fragment also lists the unstressed surfaces -- test stressors the residual architecture failed to survive -- which become the menu for the next iteration's stressor analysis.

This is the only sub-skill that produces an empirical, falsifiable claim about the SAD's quality. Per `sad/synthesis-explanation.md` §11: the Ri test is what makes residuality a method rather than an opinion. Per `shared/constitution.md` R-21 and `shared/kauffman-nkp.md` §7: a positive Ri on a held-out test set is the empirical signature that the NKP shaping in S4 + S5 successfully moved the system toward criticality.

S6 does NOT take architectural decisions. It runs the test, computes the number, and points at where the next round of stress analysis should focus.

---

## When to invoke

- `residual-design.md` exists, is approved, and the Static Architecture, Derived NFRs, and Deployment Diagram are stable.
- The architect has a fresh test stressor list -- stressors that were NOT used in S3's catalog and that did NOT inform any residue in S4 / S5.
- The user is ready to receive an empirical verdict that may be negative (Ri <= 0 means the residual architecture is no better than the naïve baseline).

## When NOT to invoke

- Residual design fragment does not exist (run S5 first).
- The user has no fresh test stressors. The empirical test cannot use the original catalog (that would not be a hold-out test). If the architect cannot generate stressors not present in the catalog, that itself is evidence the catalog is exhaustive enough for the current iteration -- but S6 still cannot run.
- The user wants to use the original Stressor Catalog as the test set "for convenience". This is structurally forbidden (train/test disjuncia per R-15-adjacent). Refuse.
- The user wants Ri to be a target ("make Ri > 0.5"). Ri is a diagnostic, not a target. Architects cannot game the test by curating it; that defeats the empirical claim.

## Pre-conditions

- The **S5 gate is marked `[x] approved`** in the project gate tracker (`FLOW.md`), not merely produced -- verify before producing (root `SKILL.md` §Gate approval protocol).
- All five prior fragments exist and are approved: `business-view.md`, `naive-architecture.md`, `flow-analysis.md`, `stressor-catalog.md`, `contagion-analysis.md`, `residual-design.md`.
- A fresh test stressor list is available, with two structural requirements:
  - The list is **disjoint** from `stressor-catalog.md` (no stressor appears in both).
  - The list is generated with the same discipline as the catalog (typed, with narratives, contextual not technical -- per `stressor-analysis` Step 3 and Step 4 requirements).

## Handoff contract

- **Consumes:** `residual-design.md` (the Y term -- residual architecture under test) + `naive-architecture.md` (the X term -- baseline control) + `business-view.md` invariants (the regression target) + a fresh, disjoint test-stressor list (S) -- prior gate S5 must be `[x] approved`.
- **Produces:** `empirical-test.md` (Ri = (Y - X) / S + unstressed surfaces + invariant-preservation check + regression note).
- **Lateral context to carry forward:** the unstressed surfaces (the next iteration's seed stressors) and a note on baseline strength (a tonto naive yields a high Ri; read the number relative to the baseline, not in absolute terms -- per the comparison-stance lesson).

## Workflow

### Step 1 -- Generate or accept the fresh test stressor list

The test list is generated by running a focused stressor session on the same business context (the Business View), but it must NOT reuse any stressor from the original catalog. Typical sources for fresh stressors:

- PESTLE / Porter / BMC / abstraction-stressing / flow-stressing run again with new prompts.
- Stressors that the team discussed and discarded during S3 (often discarded too early; the test stage is where their architectural impact is observed).
- Stressors that emerged during the S5 design conversations but did not warrant new residues at that point.
- Stressors that came up between S3 and S6 from real events (market shifts, regulatory news, technology announcements).

The minimum useful S is 5-10 stressors; smaller test sets produce Ri values too noisy to be meaningful. Larger test sets (20+) produce stronger empirical claims.

### Step 2 -- Validate train/test disjuncia

Before any application of stressors to architectures, verify the disjuncia. For every test stressor:

- The exact same stressor must not appear in `stressor-catalog.md`.
- A semantically-equivalent stressor (different wording, same triggering event + propagation chain + attractor) must not appear in `stressor-catalog.md`.
- A stressor that is a more-specific case of a catalog entry counts as a duplicate -- it cannot be in the test set.

If any test stressor fails the disjuncia check, remove it from the test set and document the removal. The architect may add a different fresh stressor in its place.

### Step 3 -- Apply each test stressor to the naïve architecture

For each test stressor in the list, evaluate the **naïve architecture** (from `naive-architecture.md`):

- Determine whether the naïve architecture survives the stressor (the system continues to operate, with degradation acceptable per the business view).
- Score `survives = 1` or `survives = 0` per stressor.
- Document the failure mode for `survives = 0` rows (which naïve component fails, what business attractor the system collapses to).

Sum the `survives` column across the test list. This is **X**.

### Step 4 -- Apply each test stressor to the residual architecture

Repeat Step 3 for the **residual architecture** (from `residual-design.md`):

- Score `survives = 1` or `survives = 0` per stressor.
- Document the rationale for `survives = 1`: which residues from the catalog cover this stressor; which Hyperliminal Coupling Map rows handle the coordination; which Topological residue protects the deployment topology.
- Document the failure mode for `survives = 0`.

Sum the `survives` column. This is **Y**.

### Step 5 -- Compute Ri

`Ri = (Y - X) / S` where S is the total test set size.

Per O'Reilly L2014-2018 (literal): `Ri = (Y - X) / S`, where `-1 <= Ri <= 1`.

Interpretation:

| Ri value | Meaning |
|---|---|
| `Ri > 0` | Positive movement toward criticality. The residual architecture survives more stress than the naïve baseline. |
| `Ri = 0` | No empirical improvement. The residual architecture is no better than the naïve baseline on this test set. Either the analysis was too shallow, the test set was poorly chosen, or the architecture's gains lie outside what this test measures. |
| `Ri < 0` | Negative. The residual architecture is WORSE than the naïve baseline on this test set. Almost always a sign of over-engineering or of residues that fight each other. Backtrack to S5 or S4. |

Per O'Reilly L2024-2026: "as Ri approaches 0 across iterations there is less and less return in doing further architectural work." Ri close to 0 in a late iteration means the architecture has stabilized; in an early iteration it means more work is needed.

#### 5.1 Read the magnitude relative to the strength of the naïve baseline

Ri is **not an absolute quality score**; its magnitude must be read against how strong the naïve baseline (X) is. The naïve architecture sometimes absorbs substantial goals already in S1 (e.g. a risk-envelope component pulled in during goal expansion), making X large. A strong baseline (high X) caps the achievable Ri: if the naïve already survives much of the test set, a modest Ri can be an excellent result -- the residual added real protection on top of an already-capable baseline. Conversely, a weak/trivial naïve (low X) inflates Ri without the residual necessarily being good in absolute terms.

Therefore: report X explicitly alongside Ri, and note the strength of the baseline in the interpretation. `Ri = 0.3` with `X = 7/10` (strong baseline) is a different and often better result than `Ri = 0.7` with `X = 0/10` (the naïve survived nothing). Do not compare Ri values across projects without also comparing baseline strength.

### Step 6 -- Identify unstressed surfaces and per-stressor regressions

Every test stressor with `residual survives = 0` is an **unstressed surface**: an area of the system the current residual architecture cannot absorb. These are the most valuable output of S6.

For each unstressed surface:

- Record the test stressor (with full narrative).
- Record the failure mode in the residual architecture (which residue is missing, which Coupling Map row should have existed, which Topological residue would have helped).
- Mark it as the starting menu for the next iteration's `stressor-analysis` (S3) -- the unstressed surface IS a stressor for the next round, with a known attractor (the failure mode observed here).

This is the connection back to S3 in the iterative flow. Per `sad/synthesis-explanation.md` §11: "Test stressors that the residual architecture fails to survive are the next round of stress analysis -- they are the unstressed surfaces of the system. The test does not say 'you are done'; it says 'here is where to stress next.'"

#### 6.1 Per-stressor regressions (residual worse than naïve)

A positive aggregate Ri can still hide a **regression**: a test stressor where the naïve survived (`naïve = 1`) but the residual does NOT (`residual = 0`). With binary survival, that is the precise signature -- distinct from a common gap where both fail (`naïve = 0, residual = 0`). A regression means a residue introduced to absorb one stressor **eroded** a capability the naïve baseline had for free.

Call these out explicitly, even when Ri is positive overall:

- Tag every row where `residual < naïve` as a **regression** (not just a generic unstressed surface).
- Name the residue (or structural decision) responsible -- the trade-off is "residue N protects against X but blocks Y".
- This is the richest evidence the test produces: it is where the architecture's own residues fight each other. Surface it for the architect to accept (documented trade-off), redesign (backtrack to S5), or re-type the offending residue.

The skill flags `Ri < 0` aggregate as a backtrack signal (Step 5); §6.1 makes sure per-stressor regressions inside a positive aggregate are not lost in the average.

#### 6.2 Invariant preservation check (the regression guard)

The Ri measures survival, not property preservation. A system can survive every test stressor and still **violate a business invariant** in the act of surviving (survive a region outage by routing data cross-border, violating a data-residency invariant). For each invariant declared in the §Business View (S1 Step 1.2), run a distinct check:

- For each invariant, ask: under any test stressor, does the **residual architecture** violate it (even on a stressor it "survives")?
- A violation is a **critical regression** -- record it in a §Invariant Preservation table with the invariant, the triggering stressor, and the residue/decision that breaks it. It is more serious than a §6.1 per-stressor regression: a non-negotiable is broken.
- This is NOT the same as "does the residual survive": survival is about continuing to operate; invariant preservation is about not breaking a declared non-negotiable while doing so.

If the §Business View declares no invariants, this step is empty (do not invent them -- R-09). The check exists so the empirical test answers both questions: criticality (survival) AND non-negotiable preservation. The decomposition is still residue-driven; this is acceptance verification, not design (R-01 / R-21).

### Step 7 -- Hygiene checks before writing the fragment

| Check | Verification | Rule |
|---|---|---|
| Test set disjoint from catalog | No test stressor (or semantic equivalent) appears in `stressor-catalog.md`. | R-15-adjacent (train/test split) / synthesis decision |
| Each test stressor has a narrative | Same Step-3 narrative requirement as S3. Cause -> propagation -> attractor. | S3 doctrine carried into S6 |
| No probability or cost on the test set | No filtering, weighting, or ranking by likelihood / cost. | R-20 |
| X and Y are integers | Survival is binary; partial credit defeats the test. | -- |
| S = test set size | The denominator is the COUNT of test stressors, not weighted. | -- |
| Ri reported with the formula | `Ri = (Y - X) / S = <value>` written explicitly. | O'Reilly L2014-2018 |
| **X / Y / S / Ri reconcile (script-backed)** | After drafting and **before** parking the fragment, run the bundled `scripts/fragment-checks/check_counts.py <fragment>`. X = count of `Naive survives` = yes/1 rows; Y = count of `Residual survives` = yes/1 rows; S = row count; and the parenthesised `Ri = (Y - X) / S` must match the table. The script hard-fails on drift -- never hand-carry X/Y/S into prose without re-deriving from the survives table. | counts reconcile (feedback F-02) |
| Unstressed surfaces enumerated | Every `residual survives = 0` is documented. | S6 Step 6 |
| Unstressed surfaces marked for next iteration | Each surface is annotated as a next-round stressor candidate. | Iterative flow |
| Regressions flagged | Every row where `naïve = 1` and `residual = 0` is tagged as a regression with the responsible residue named, even when aggregate Ri > 0. | S6 Step 6.1 |
| Baseline strength noted | X is reported alongside Ri and the interpretation notes whether the naïve baseline is strong or trivial. | S6 Step 5.1 |
| Invariants verified | Every invariant declared in the §Business View has been checked against the residual under the test stressors; violations are recorded as critical regressions. Empty only if the Business View declares no invariants. | S6 Step 6.2 |

If a deterministic check fails, refuse to write the fragment.

## Output contract

One fragment file in the project workspace:

### `empirical-test.md`

Markdown fragment that fills SAD §Empirical Test (Residual Index). Structure:

1. One paragraph orienting the reader: iteration number, test set size S, naïve survival X, residual survival Y, Ri value.
2. **Test stressor table** -- columns: `#` / `Test Stressor (narrative)` / `Naïve survives?` / `Residual survives?` / `Rationale`.
3. **Ri computation** -- formula with values substituted: `Ri = (Y - X) / S = (<Y> - <X>) / <S> = <Ri value>`.
4. **Interpretation** -- one paragraph: positive movement / no improvement / negative, read relative to the strength of the naïve baseline X (Step 5.1).
5. **§Unstressed Surfaces** -- the rows where `residual survives = 0`, with failure modes and next-iteration stressor candidates.
6. **§Regressions** (if any) -- the rows where `naïve = 1` and `residual = 0`, each naming the residue responsible for the trade-off (Step 6.1). Omit the section only if there are no regressions.
7. **§Invariant Preservation** (if the Business View declares invariants) -- table: `Invariant` / `Preserved under all test stressors?` / `Violating stressor + residue (if any)`. A violation is a critical regression (Step 6.2). Omit only if no invariants were declared.
8. **§Iteration history table** (if this is iteration N >= 2) -- previous Ri values to track convergence.

No frontmatter.

## Refusal conditions

| # | Trigger | Rule | Returned message |
|---|---|---|---|
| 1 | `residual-design.md` does not exist or is unapproved. | Pre-condition | Refuse. Direct the user to run / approve `residual-design` (S5) first. |
| 2 | The test set is empty or has fewer than 5 stressors. | -- | Heuristic warning, not a hard refusal. Ri values from very small test sets are too noisy to be informative. Recommend the architect add stressors before proceeding. |
| 3 | Any test stressor appears verbatim in `stressor-catalog.md`. | Train/test disjuncia | List the offending stressor. The test set must be disjoint. Remove from test set. |
| 4 | A test stressor is semantically equivalent to a catalog entry (same triggering event + propagation + attractor, different wording). | Train/test disjuncia | List the offending pair. Same as #3 with judgment call. |
| 5 | The test set contains a Probability / Likelihood / Cost / Risk column. | R-20 | Remove the column. The test set is generated under the same discipline as the catalog. |
| 6 | A test stressor is a one-liner without narrative. | S3 doctrine carried over | Reject. Demand the narrative (cause -> propagation -> attractor) per S3 Step 3. |
| 7 | A test stressor is technical-not-contextual ("Redis goes down"). | S3 doctrine carried over | Reject. Reframe contextually per S3 Step 4. |
| 8 | X or Y is reported as a non-integer (partial credit). | Method | Reject. Survival is binary. A stressor either is absorbed (1) or it is not (0). Partial credit defeats the empirical claim. |
| 9 | Ri is reported but unstressed surfaces are NOT enumerated. | S6 Step 6 | Reject. The Ri number alone is half the output. Unstressed surfaces are the menu for the next iteration. |
| 10 | Ri is negative AND the user wants to ship without addressing it. | -- | Heuristic warning. Document the negative Ri explicitly and recommend backtracking to S4 / S5. Do not silently accept a negative Ri as success. |
| 11 | The user requests that Ri be presented as a target metric (e.g., "make Ri > 0.7"). | Method | Refuse the framing. Ri is a diagnostic, not a target. Gaming the test by curating it defeats the empirical claim. |

## Worked example

See `sad/examples/ev-charging-sad.md` §9 (Empirical Test) for a complete worked example. EV charging's iteration-1 test:

| # | Test Stressor | Naïve survives? | Residual survives? |
|---|---|---|---|
| T1 | Charger firmware vulnerability requires emergency offline patch | 0 | 1 |
| T2 | New regulation: real-time energy reporting to grid operator | 0 | 1 |
| T3 | Customer wants to pay via fleet operator (B2B billing) | 0 | 1 |
| T4 | Vandalism wave in a specific city | 0 | 1 |
| T5 | Solar/wind smart-grid pricing changes electricity cost every 15 min | 0 | 0 |
| T6 | Charger needs to reverse-feed power to grid (V2G) | 0 | 0 |

- X = 0, Y = 4, S = 6
- **Ri = (4 - 0) / 6 ~ 0.67**
- Interpretation: positive movement toward criticality.
- **Unstressed surfaces:** T5 (dynamic energy pricing) and T6 (V2G). Next iteration's stressor analysis starts from these two surfaces.

T1-T4 trace back to existing residues in the SAD (residue #1 polymorphic charger interface; auth/billing decoupling residue #3; sensor + camera residues #8; degraded mode residue #9). T5 and T6 require new residues that the current iteration did not produce.

## Why these rules

- **Train/test disjuncia.** The empirical claim of residuality rests on a held-out test set. Using the original catalog as the test set would be like training a machine learning model on the test data: the result is meaningless. Per O'Reilly L2036-2040 (the ML analogy), the architectural discipline borrows the same rigor.
- **No probability / cost on the test set.** The catalog is generated without probability filtering (R-20); the same discipline applies to the test set. If the architect filters the test set by "what is likely to happen", the test loses its random-simulation character.
- **Binary survival.** Partial credit defeats the test. A system either survives a stressor (continues to operate per the business view's acceptable degradation) or it does not. The architect's intuition about "kind of survives" is exactly what residuality is designed to displace with an empirical test.
- **Ri is not a target.** Per O'Reilly L2024-2026: as Ri approaches 0 across iterations, further architectural work yields diminishing returns. The number is a diagnostic for "are we done iterating?", not a KPI to optimize.
- **Unstressed surfaces are the bridge to the next iteration.** Without them, S6 produces a number and stops. The next iteration's S3 needs the failure modes; documenting them in S6 is what makes the meta-skill iterative.
